BreachLab Backend — Roadmap
===========================

What's actually left to do, ranked by whether it matters now,
post-launch, or not at all. Most of the original wishlist was either
covered by Polar/Clerk/Sentry or premature.


===================================================================
A. Build reactively when real users surface the need
===================================================================

  - Per-domain leaderboards (APPSEC, SOC, NETWORK, WIRESHARK, GRC)
    — build when users ask.
  - Integration tests for payment + auth flows — write when something
    breaks in prod.
  - Account deletion self-serve (GDPR) — only matters with EU users.
    Clerk delete + DB cascade already handles the basic case.
  - Resume in-progress attempt — only if drop-off data shows users
    abandoning long challenges.
  - Ban/disable user — only when moderation is a real problem.
  - View a single user's full submission history (admin) — when first
    support ticket needs it.


===================================================================
B. Skip — vendor-covered or premature
===================================================================

Already handled by services you're already paying for:
  - Transactional email (welcome, receipts, payment failed,
    email-change confirmations, webhook-failure alerts) — Polar +
    Clerk + Sentry already do all of these.
  - Refund subscription on user's behalf — Polar dashboard.
  - DB backups — your managed DB host does this. Verify in their
    dashboard and write a one-page restore runbook. No code.

Premature:
  - Per-difficulty / time-windowed leaderboards.
  - Materialized view at 1k users (ZSET handles 100k+ fine).
  - Switch rate limiter to Redis store (only matters when running
    multiple instances).
  - Bulk import challenges (write a one-off script the day you need it).
  - Audit log of admin actions (Sentry breadcrumbs + Prisma updatedAt
    covers most debugging).
  - S3 multipart / lifecycle / orphan cleanup — challenge files are
    small zips, S3 bill is rounding error.
  - Grader extraction with regex/numeric/range matchers — only if you
    want richer question types. Current exact-match works.
  - OpenAPI spec — only if your frontend team is asking.

Feature ideas (not gaps):
  - Tags on challenges.
  - Hints (purchasable / free).
  - Walkthroughs unlocked after passing.
  - Challenge prerequisites.
  - Per-question points instead of evenly splitting challenge.points.
  - Bookmarks / favorites.
  - Achievements / badges (badges: String[] exists on User but unused
    — either wire it up or drop the column).
  - Per-user notifications.
  - Friends / following / activity feed.

These are product decisions, not infrastructure gaps. Build when
they're product-shaped.


===================================================================
C. Environmental — not features, just stuff to chase
===================================================================

  - Resolve local DB connection (prod works, local times out).
  - Verify .github/workflows/deploy.yml runs typecheck + lint on PR.
    5-min audit; add the steps if missing.


===================================================================
Already shipped
===================================================================

Pre-launch core (was section A):
  - /health pings Redis + DB with ok/degraded/down states.
  - Auto-rebuild leaderboard on empty ZSET at startup.
  - Admin: edit individual questions on a challenge
    (POST/PATCH/DELETE on /challenges/:id/questions[/:questionId]).
  - Admin: grant/revoke ADMIN role + change PRO/FREE subscription.
  - Leaderboard recompute on challenge points change or soft-delete.

Earlier infrastructure:
  - Subscriptions (checkout, webhooks with idempotency, portal,
    FREE↔PRO flips, subscriptionEndsAt).
  - Global leaderboard (Redis ZSET, paginated read with `me` block,
    admin rebuild, live updates on submit/profile/delete).
  - Rate limiting (submit, download, webhook with IPv6-safe per-user
    keys).
  - Redis client (shared ioredis, connect-on-startup).
  - Sentry (init in instrument.ts, express handler).
  - Graceful shutdown (SIGTERM/SIGINT).
  - 404 JSON fallback.
  - prisma migrate deploy.
  - Public challenge browse + anonymous detail view.
  - Public profile endpoints.
  - Submissions tab data + per-submission report.
